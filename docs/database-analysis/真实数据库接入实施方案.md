## 一、结论

当前方案可以进入评审，但开工前必须以本文作为准入基线：

| 项目 | 结论 | 说明 |
|------|------|------|
| 接入范围 | 可以开工 | P0 固定 10 张源表，先支撑首页、任务、设备、盘位、容量、告警、用户展示 |
| 技术路线 | 可以开工 | 源站点 MySQL → 同步服务 → PostgreSQL 中心库 → API → 前端 |
| 大表策略 | 可以开工 | `tbl_file`、`tbl_folder`、`tbl_zip_file` 等不做全量同步 |
| 风险项 | 需修正后开工 | 部分源表无更新时间字段，不能统一按 `update_dt` 增量 |

## 二、P0 接入范围

| 领域 | 源表 | 主键 | 同步策略 | 页面用途 |
|------|------|------|----------|----------|
| 任务 | `tbl_task` | `id` | `update_dt` 增量 + 首次分页全量 | 首页统计、任务管理 |
| 设备 | `tbl_disc_lib` | `lib_id` | 小表全量快照 + 差异 upsert | 首页设备、盘架管理 |
| 盘位 | `tbl_slots` | `slot_id` | 中表全量快照 + 差异 upsert | 盘位视图、容量统计 |
| 盘笼 | `tbl_magzines` | `mag_id` | 小表全量快照 + 差异 upsert | 盘笼位置、离在线状态 |
| 光驱 | `tbl_drivers` | `driver_id` | 小表全量快照 + 差异 upsert | 设备详情 |
| 硬盘 | `tbl_hd_info` | `slot_id` | 小表全量快照 + 差异 upsert | 硬盘健康度 |
| 存储卷 | `tbl_logical_volume` | `volume_id` | `update_time` 增量 | 容量统计 |
| 告警 | `tbl_early_warning` | `id` | `create_date` 增量 + 状态回补 | 告警统计、告警列表 |
| 分组 | `tbl_lib_group` | `id` | 静态/小表全量快照 | 设备分组 |
| 用户 | `tbl_user` | `user_id` | `update_time` 增量 | 用户与权限 |

说明：
- `tbl_disc_lib`、`tbl_slots`、`tbl_magzines`、`tbl_drivers`、`tbl_hd_info`、`tbl_lib_group` 没有可靠 `update_dt` 字段，不能按统一时间游标处理。
- 小表优先采用全量快照 + hash 差异写入，降低源库改造成本。
- 如后续允许改源库，可新增 `update_dt` 后切换为增量。

## 三、中心库 Schema 基线

### 3.1 通用原则

统一中心库使用 PostgreSQL。所有同步表必须包含：

| 字段 | 说明 |
|------|------|
| `id` | 中心库自增主键 |
| `source_site_id` | 来源站点 |
| `source_table` | 来源表 |
| `source_id` | 来源主键 |
| `source_hash` | 源记录关键字段 hash，用于无更新时间表差异判断 |
| `source_updated_at` | 源表更新时间，没有则为空 |
| `synced_at` | 本次同步时间 |
| `raw_data` | 原始数据 JSONB，便于回查 |

唯一约束统一为：

```sql
UNIQUE(source_site_id, source_table, source_id)
```

### 3.2 基础表


## 五、同步服务实现口径

### 5.1 同步模式

| 模式 | 适用表 | SQL |
|------|--------|-----|
| incremental | `tbl_task`、`tbl_logical_volume`、`tbl_user` | `WHERE cursor > last_sync_time ORDER BY cursor, pk LIMIT ?` |
| append_incremental | `tbl_early_warning` | `WHERE create_date > last_sync_time OR id > last_sync_id` |
| snapshot | `tbl_disc_lib`、`tbl_slots`、`tbl_magzines`、`tbl_drivers`、`tbl_hd_info`、`tbl_lib_group` | `SELECT ... ORDER BY pk LIMIT ? OFFSET ?` |

### 5.2 Upsert 规则

1. 每条源记录计算 `source_hash`。
2. 中心库不存在则插入。
3. 中心库存在且 hash 不同则更新。
4. 中心库存在且 hash 相同则跳过。
5. 同步成功后更新 `sync_progress`。

### 5.3 删除/离线处理

| 场景 | 处理 |
|------|------|
| 源表有删除标识 | 同步删除标识，不物理删除中心库数据 |
| snapshot 表本轮未出现 | 标记 `missing_in_last_snapshot = true`，连续 3 轮缺失再标记失效 |
| 设备离线 | 以 `device_status` 为准，不删除 |
| 用户过期 | 以 `expiration_time` 为准，不删除 |

## 六、API Contract

第一期开只读接口，前端先替换 Mock 数据来源。

| 接口 | 参数 | 返回 |
|------|------|------|
| `GET /api/sites` | - | 站点列表、同步状态 |
| `GET /api/dashboard/summary` | `siteId?` | 任务、设备、容量、告警统计 |
| `GET /api/tasks` | `siteId? status? page pageSize` | 任务分页列表 |
| `GET /api/tasks/:id` | - | 任务详情 |
| `GET /api/racks` | `siteId? status?` | 设备/盘架列表 |
| `GET /api/racks/:id/slots` | - | 盘位列表 |
| `GET /api/alerts` | `siteId? level? status? page pageSize` | 告警列表 |
| `GET /api/users` | `siteId? keyword? page pageSize` | 用户列表 |
| `GET /api/sync/status` | `siteId?` | 各站点各表同步状态 |

统一返回格式：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "traceId": "20260528-xxxx"
}
```

分页返回格式：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 100
}
```

## 七、安全与部署

| 项目 | 要求 |
|------|------|
| 源库账号 | 只读账号，禁止 DDL/DML |
| 密码存储 | 不进代码库，使用环境变量或密钥管理 |
| 网络 | 同步服务到源库白名单访问 |
| 日志 | 不打印密码、token、源库连接串 |
| 数据权限 | API 按用户可访问站点过滤 `source_site_id` |
| 审计 | 所有写操作后续进入审计日志；一期只读接口记录查询日志即可 |

## 十、暂不纳入一期

| 内容 | 原因 | 后续方案 |
|------|------|----------|
| `tbl_file` 全量同步 | 数据量过大 | Elasticsearch/按需查询 |
| 文件内容同步 | 不是统一平台定位 | 流式读取 |
| 写回源站点 | 风险高 | 二期评估 |
| 跨站点任务调度 | 涉及业务闭环 | 等只读接入稳定后再做 |
| 权限写入同步 | 影响生产用户 | 一期只读展示 |
