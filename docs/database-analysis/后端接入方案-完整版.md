# 后端接入方案

> 版本: v1.0
> 日期: 2026-05-28
> 目标: 前端保持稳定，不因真实数据库字段变化而大改

---

## 一、架构概览

```
源站点 MySQL (只读)
    ↓
同步服务 (定时/增量)
    ↓
PostgreSQL 中心库 (统一视图)
    ↓
API Layer (Node.js/Go)
    ↓
前端 Provider (只换数据源)
```

**核心原则**: 前端 Provider 接口不变，Adapter 层做字段映射。

### 3.4 raw_data JSONB 使用说明

```sql
raw_data JSONB NOT NULL  -- 每个 unified_* 表都有此字段
```

**用途**:
- 保留源表完整原始数据，用于数据追溯和回查
- Adapter 层可从 raw_data 读取未映射字段

**约束**:
- **不作为前端直接读取字段**
- 前端只通过 Adapter 映射后的字段访问数据
- raw_data 是内部追溯用，不暴露给客户端

**P2 阶段评估**:
- 如果 raw_data 造成存储膨胀，可考虑裁剪
- 或仅保留变更记录的 raw_data
- 或转为可选字段（nullable）

## 四、P0 表同步映射

| 源表 | 中心库表 | 同步策略 | 关键字段映射 |
|------|----------|----------|--------------|
| `tbl_task` | `unified_tasks` | `update_dt` 增量 | `task_type`→`type`, `status`→`status` |
| `tbl_disc_lib` | `unified_devices` | 全量快照+hash差异 | `lib_id`→`rackId`, `IP`→`ip` |
| `tbl_slots` | `unified_slots` | 全量快照+hash差异 | `serial_num`→`discNo` |
| `tbl_magzines` | `unified_magazines` | 全量快照+hash差异 | `mag_id`→`magId`, `rfid`→`rfid` |
| `tbl_drivers` | `unified_drivers` | 全量快照+hash差异 | `driver_sn`→`serialNumber` |
| `tbl_hd_info` | `unified_hard_disks` | 全量快照+hash差异 | `serial_num`→`serialNumber` |
| `tbl_logical_volume` | `unified_volumes` | `update_time` 增量 | `type`→`volumeType` |
| `tbl_early_warning` | `unified_alerts` | `create_date` 增量 | `s_level`→`severity` |
| `tbl_lib_group` | `unified_device_groups` | 全量快照 | `group_id`→`groupId` |
| `tbl_user` | `unified_users` | `update_time` 增量 | `name`→`displayName` |

**同步模式说明**:
- `增量`: 有可靠 `update_dt`/`update_time` 字段，按时间戳增量
- `全量快照+hash差异`: 无可靠更新字段，全量拉取后比对 hash 判断变更

### 4.1 source_hash 使用原则

```
源表有 update_dt/update_time 字段?
    ├─ 是 → 优先使用时间戳增量，source_hash 可选
    └─ 否 → 使用 hash 比对判断变更
```

**不强制**第一阶段所有表计算 hash。优先确认源表是否有更新时间字段。

### 4.2 一期同步策略（不实现 Staging）

```
Sprint 1: 直接 UPSERT 到 unified_* mirror 表
Sprint 2+: 按需引入 Staging Area 作为增强
```

**理由**: Staging 层增加复杂度，P0 阶段直接 upsert 更简单。

## 五、DTO / Adapter 设计

### 5.1 设计原则

```
前端类型 (lib/types/*)
    ↑
    ↓ (Adapter 做字段映射)
API 响应 DTO
    ↑
    ↓
中心库统一表
```

### 5.2 任务 Adapter 示例

```typescript
// lib/api/adapters/task-adapter.ts

import type { TaskItem } from "@/lib/types/task"
import type { UnifiedTask } from "./dto-types"

export function adaptTask(source: UnifiedTask): TaskItem {
  return {
    id: String(source.source_id),
    name: source.task_name || `任务-${source.source_id}`,
    taskNo: source.task_uuid || String(source.source_id),
    type: adaptTaskType(source.task_type),
    phase: adaptPhase(source.status),
    status: adaptStatus(source.status),
    priority: adaptPriority(source.priority),
    // ... 其他字段映射
    archiveName: source.raw_data?.archive_name || "",
    siteCode: source.source_site_id,
    // ...
  }
}

function adaptTaskType(type: number): TaskItem["type"] {
  const MAP: Record<number, TaskItem["type"]> = {
    1: "full_package",
    2: "incremental_package",
    3: "backup",
    4: "restore",
    5: "full_scan",
    6: "incremental_scan",
    // ...
  }
  return MAP[type] || "other"
}

function adaptPhase(status: number): TaskItem["phase"] {
  // 状态码到阶段的映射
  if (status === 0) return "pending"
  if (status === 1) return "scanning"
  // ...
  return "pending"
}
```

### 5.3 盘架 Adapter 示例

```typescript
// lib/api/adapters/rack-adapter.ts

import type { Rack, RackSlot } from "@/lib/types/rack"
import type { UnifiedDevice, UnifiedSlot, UnifiedMagazine } from "./dto-types"

export function adaptRack(
  device: UnifiedDevice,
  slots: UnifiedSlot[],
  magazines: UnifiedMagazine[]
): Rack {
  return {
    id: String(device.source_id),
    rackId: String(device.lib_id),
    rackName: device.name || `设备-${device.lib_id}`,
    siteCode: device.source_site_id,
    ip: device.ip,
    totalSlots: device.slots,
    usedSlots: slots.filter(s => s.disc_type > 0).length,
    usagePercent: Math.round((slots.filter(s => s.disc_type > 0).length / device.slots) * 100),
    status: adaptDeviceStatus(device.device_status),
    slots: slots.map(adaptSlot),
    // ...
  }
}

function adaptDeviceStatus(status: number): Rack["status"] {
  const MAP: Record<number, Rack["status"]> = {
    0: "normal",
    1: "warning",
    2: "fault",
    3: "maintenance",
  }
  return MAP[status] || "normal"
}
```

## 六、Mock Provider 替换顺序

### 6.1 替换原则

**不改变前端类型定义，只在 Adapter 层做字段转换。**

### 6.2 运行时切换机制

```typescript
// lib/api/index.ts
const mode = process.env.NEXT_PUBLIC_API_MODE || "mock"

// NEXT_PUBLIC_API_MODE 值:
// - "mock": 使用 mock provider（默认，demo 环境）
// - "api": 使用真实 API（生产/集成环境）
```

### 6.3 Mock Fallback 策略

API 请求失败时自动降级到 Mock，不影响 demo 页面展示：

```typescript
// lib/api/real/base-client.ts
async function withMockFallback<T>(
  apiFn: () => Promise<T>,
  mockFn: () => T,
  operation: string
): Promise<T> {
  try {
    return await apiFn()
  } catch (error) {
    console.warn(`[API Fallback] ${operation} failed, using mock:`, error)
    return mockFn()
  }
}

// 使用示例
const tasks = await withMockFallback(
  () => fetchTasks(filters),
  () => mockTaskProvider.getAll(filters),
  "getTasks"
)
```

**降级规则**:
- `NEXT_PUBLIC_API_MODE=mock`: 始终使用 mock，不发起 API 请求
- `NEXT_PUBLIC_API_MODE=api`: 优先调用 API，失败时降级到 mock
- 降级时 console.warn 但不影响页面正常渲染

### 6.4 P0 不接入的 Provider

| Provider | 状态 | 说明 |
|----------|------|------|
| `AuditProvider` | 暂保留 mock | P0 不接真实审计接口 |
| `SettingsProvider` | 暂保留 mock | P0 不接真实设置接口 |
| 其他 Provider | 可接入 | 按 Sprint 3 计划逐步替换 |

**理由**: 审计日志和系统设置非核心展示功能，P0 阶段不影响 demo 验证。

### 6.5 替换顺序

```
阶段1: 认证 API
  └→ lib/api/auth.ts (新增)
      └→ lib/api/adapters/auth-adapter.ts

阶段2: 站点 + 首页统计
  └→ lib/api/sites.ts
      └→ lib/api/adapters/site-adapter.ts
  └→ lib/api/dashboard.ts
      └→ lib/api/adapters/dashboard-adapter.ts

阶段3: 任务列表 (P0 表)
  └→ lib/api/tasks.ts
      └→ lib/api/adapters/task-adapter.ts
  └→ lib/api/providers.ts (TaskProvider 实现)

阶段4: 盘架/盘位 (P0 表)
  └→ lib/api/racks.ts
      └→ lib/api/adapters/rack-adapter.ts
  └→ lib/api/providers.ts (RackProvider 实现)

阶段5: 告警 + 用户
  └→ lib/api/alerts.ts
  └→ lib/api/users.ts

阶段6: 日志 + 检索
  └→ lib/api/audits.ts
  └→ lib/api/search.ts

**注意**: AuditProvider 和 SettingsProvider 在 P0 阶段不替换，暂保留 mock。

### 7.2 检索方案

```
前端检索请求
    ↓
API Gateway
    ↓
┌─────────────────────────────────┐
│  PostgreSQL (P0 表)              │ ← 设备、任务、用户、告警
│  毫秒级响应                       │

    ↓ (跨表检索时)
┌─────────────────────────────────┐
│  Elasticsearch                   │ ← 文件索引检索
│  模糊匹配、分词、聚合              │

### 8.5 状态码枚举风险

Adapter MAP 必须覆盖所有源状态码，未识别状态码必须 fallback 到安全值：

| 源字段 | 枚举类型 | 必须覆盖 |
|--------|----------|----------|
| `tbl_task.status` | `TaskPhase` | pending/scanning/completed/failed/paused/idle |
| `tbl_disc_lib.device_status` | `Rack["status"]` | normal/warning/fault/maintenance |
| `tbl_slots.disc_type` | `RackSlot["mediaType"]` | hdd/bd/offline |
| `tbl_logical_volume.type` | `StorageVolume["type"]` | optical/magnetic/composite |

**规则**:
```typescript
// 未识别状态码必须 fallback，不允许页面崩溃
function adaptStatus(status: number): string {
  return MAP[status] ?? "unknown"  // 永远有 fallback
}
```
