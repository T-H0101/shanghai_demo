## 1. P0 表数量评估

**结论: 10 张表**数量合理。

| 评估维度 | 结论 |

| 覆盖首页统计 | ✅ 任务数、设备数、容量、告警数均可支撑 |

## 6. DTO/Adapter 前端类型不变保证

**结论: 设计合理，可保证前端类型不变。**

| 机制 | 实现 | 状态 |
|------|------|------|
| 类型契约 | `lib/types/*` 不变 | ✅ |
| Adapter 层隔离 | `lib/api/adapters/*` | ✅ |
| MAP 映射 | 状态码→枚举，ID→String | ✅ |
| 缺失字段兜底 | `|| "默认值"` | ✅ 示例有 |

**风险点**:
- Adapter 中的 MAP 必须穷举所有源状态码，否则 fallback 到 `"other"`/`"pending"`
- 需要与后端确认源表状态码枚举值

## 7. Mock Fallback 明确性

**结论: 部分明确，需加强。**

| 机制 | 位置 | 状态 |
|------|------|------|
| 环境变量切换 | `NEXT_PUBLIC_API_MODE` | ✅ 方案中提及 |
| Provider 选择逻辑 | `lib/api/index.ts` | ⚠️ 方案中提及，但代码未实现 |
| Mock 数据保留 | `lib/mock/*` | ✅ |
| Error fallback | API 失败时降级 Mock | ❌ **未说明** |

**建议补充**:
```typescript
// API 请求失败时的降级策略
async function withMockFallback<T>(apiFn: () => Promise<T>, mockFn: () => T): Promise<T> {
  try {
    return await apiFn()
  } catch (error) {
    console.warn('API failed, falling back to mock:', error)
    return mockFn()
  }
}
```

## 10. 过度设计与实现风险

### 潜在过度设计

| 项 | 风险 | 建议 |
|----|------|------|
| `raw_data JSONB` 全量存储 | 存储膨胀，日后难以维护 | P0 可保留，P2 评估价值后裁剪 |
| `source_hash` 字段 | hash 计算开销 | 确认是否必需，或用 `updated_at` 比对 |
| Staging Area | 增加复杂度 | 一期直接 UPSERT 即可 |

### 实现风险

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 源表字段名不一致 | **高** | 需逐表核对 `schema-inventory.json` |
| 状态码枚举不统一 | **高** | Adapter MAP 必须可配置 |
| 无 `update_dt` 字段 | **中** | snapshot + hash 方案已规划 |
| 多站点 JOIN 查询性能 | **中** | 已有索引设计，需压测验证 |
| 前端类型变更 | **低** | 严格遵循禁止修改类型定义 |
