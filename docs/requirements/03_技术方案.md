## 2. 核心框架说明

### 2.1 Next.js 16（App Router）

**作用**：
- 页面路由：`app/` 目录下的文件自动映射为路由
- 布局系统：共享布局 + 嵌套布局
- 静态预渲染：所有页面默认静态预渲染（`○ Static`）
- 服务端组件：默认服务端组件，客户端组件通过 `"use client"` 声明

**在本项目中的使用**：
- 所有页面均为客户端组件（`"use client"`）
- 使用 `AppShell` 组件实现统一布局
- 使用 `generateStaticParams` 预留动态路由能力（当前未使用）

**技术取舍**：
- 选择 App Router 而非 Pages Router，因为 App Router 支持布局嵌套、更现代
- 选择静态预渲染因为 Demo 阶段不需要服务端数据

### 2.2 React 19

**作用**：
- 组件化开发
- Hooks（`useState`、`useEffect`、`useCallback` 等）
- 客户端状态管理

**在本项目中的使用**：
- 页面级状态使用 `useState` + `useCallback`
- 跨组件通知使用模块级单例 + `useEffect` 订阅模式
- 所有交互逻辑使用 React Hooks 实现

**技术取舍**：
- 使用 Hooks 而非类组件，代码更简洁
- 状态管理轻量级，不引入 Redux（Demo 阶段不需要）

### 2.3 TypeScript

**作用**：
- 类型安全：定义 mock 数据类型、组件 props 类型
- 类型推导：减少 any 类型使用
- IDE 支持：自动补全、类型检查

**在本项目中的使用**：
- `lib/types/` 目录下定义所有业务类型
- 组件 props 使用 TypeScript 定义
- 使用 `interface` 和 `type` 定义枚举和联合类型

**技术取舍**：
- TypeScript 的 `skipLibCheck: true` 和 `ignoreBuildErrors: true` 允许构建跳过类型检查，因为 Demo 阶段 mock 数据类型可能与真实 API 类型不一致
- 生产环境应关闭这两个选项，确保类型安全

### 2.4 Tailwind CSS v4

**作用**：
- 原子化 CSS：避免样式冲突
- 响应式设计：`sm:`、`md:`、`lg:` 前缀
- 暗色模式支持（通过 `class` 或 `data` 属性）

**在本项目中的使用**：
- 所有页面使用 Tailwind CSS 类名
- 使用 `@tailwindcss/postcss` 作为 PostCSS 插件
- 颜色系统基于 Slate、Gray、Blue 等中性色

**技术取舍**：
- 选择 Tailwind 而非 CSS Modules，因为 Tailwind 更适合快速原型开发
- 使用 slate 灰蓝色系，配合蓝色强调色，符合企业软件风格

### 2.5 shadcn/ui 设计模式

**作用**：
- 基于 Radix UI 的无障碍组件封装
- 一致的设计语言和交互模式
- 可复制可定制的组件源码

**在本项目中的使用**：
- `components/ui/` 目录下 57 个 UI 组件
- 使用 `cn()` 工具函数合并类名
- 使用 Radix UI 原语（Primitive）构建复杂组件

**组件清单**（按类别）：
| 类别 | 组件 |
|------|------|
| 数据展示 | Card、Table、Badge、Progress、Skeleton |
| 表单 | Input、Select、Switch、Checkbox、RadioGroup、Slider、Textarea |
| 反馈 | Toast、Alert、Dialog、Drawer、Sheet |
| 导航 | Tabs、Breadcrumb、Menu、Sidebar、Pagination |
| 布局 | Resizable、ScrollArea、Separator、Collapsible |
| 数据选择 | Calendar、DatePicker、Popover、Tooltip、Command |

**技术取舍**：
- shadcn/ui 不是传统意义的"库"，而是可复制粘贴的源码，适合定制化需求
- Demo 阶段直接使用，保持与生产环境一致的代码风格

### 3.2 Mock 数据特点

| 特点 | 说明 |
|------|------|
| 静态数据 | 所有数据为 `const` 导出，页面刷新后恢复初始值 |
| 类型安全 | 每个 mock 文件对应 `lib/types/` 下的类型定义 |
| 真实感 | 数据内容符合业务场景（临床试验、影像数据等） |
| 数据量有限 | 仅用于展示和交互演示，不适合性能测试 |

### 3.3 Mock 数据的使用方式

**直接在页面组件中 import**：
```typescript
import { sites } from "@/lib/mock/sites"
import { taskStats, tasks } from "@/lib/mock/tasks"
```

**配合 useState 管理本地状态**：
```typescript
const [tasks, setTasks] = useState<TaskItem[]>(mockTasks)
```

## 4. 前端交互方案

### 4.1 状态管理策略

| 层级 | 方案 | 适用场景 |
|------|------|----------|
| 页面级状态 | `useState` | 列表筛选、分页、Tab 切换等 |
| 跨组件状态 | 模块级单例 + `useEffect` | 通知中心 |
| 表单状态 | `useState` + `onChange` | 设置页面表单 |
| 全局状态 | 未使用（Demo 不需要） | — |

**模块级单例通知方案**（`store/notification.ts`）：
```typescript
let notifState = { notifications: [...], unreadCount: 3 }
type Listener = (state: NotificationState) => void
const listeners: Set<Listener> = new Set()

export function useNotificationStore() {
  const [state, setState] = React.useState<NotificationState>({...notifState})
  React.useEffect(() => {
    const listener: Listener = (newState) => setState({...newState})
    listeners.add(listener)
    listener({...notifState})
    return () => { listeners.delete(listener) }
  }, [])
  // ...
}
```

**技术取舍**：
- 未引入 Zustand/Jotai/Redux，因为 Demo 阶段状态管理不复杂
- 模块级单例适合通知中心这种"全局但轻量"的状态

### 4.2 Toast 反馈方案

**使用 `hooks/use-toast.ts`**：
```typescript
import { toast } from "@/hooks/use-toast"

toast({ title: "保存成功", description: "系统设置已保存" })
toast({ title: "导出失败", variant: "destructive" })
```

**Toast 类型**：
- `default`：默认样式（蓝色边框）
- `destructive`：错误/警告样式（红色边框）

### 4.3 Dialog 方案

## 5. 后续真实后端接入思路

### 5.1 API 层设计建议

```typescript
// lib/api/request.ts
const BASE_URL = process.env.NEXT_PUBLIC_API_URL || '/api'

export async function request<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  const res = await fetch(`${BASE_URL}${endpoint}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${getToken()}`,
      ...options?.headers,
    },
  })
  if (!res.ok) throw new ApiError(res.status, await res.text())
  return res.json()
}
```

### 5.2 数据获取建议

| 场景 | 推荐方案 |
|------|----------|
| 列表数据 | React Query（支持缓存、分页、轮询） |
| 单次查询 | SWR（轻量） |
| 表单提交 | 直接 fetch + 乐观更新 |
| 实时数据 | WebSocket / SSE |

### 5.3 状态管理建议

| 场景 | 推荐方案 |
|------|----------|
| 页面级状态 | useState + useReducer |
| 跨组件状态 | Context API / Zustand |
| 服务端状态 | React Query |
| 表单状态 | React Hook Form |
