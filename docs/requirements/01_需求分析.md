# 统一光盘库管理平台 - 需求分析

## 1. 项目背景

**项目名称**：统一光盘库管理平台（Disc Library Management Platform）

**项目阶段**：前端 Demo + Mock 数据 + 前端交互（P0 已完成）

本项目是药明康德企业级光盘库管理系统的**前端演示版本**，旨在展示统一管控多数据中心光盘库的交互能力。系统通过标准化接口对接各站点系统，实现统一视图、统一管控、统一权限、统一任务调度，解决多站点独立部署带来的数据割裂、管控分散问题。

> ⚠️ 本文档描述的仅为 Demo 原型的需求分析，不构成生产系统规格说明。

### 4.2 核心功能模块详情

#### 4.2.1 站点管理（2.1）

**功能**：
- 统一管理所有站点基础信息（名称、编码、IP、端口、状态、联系人）
- 支持站点新增、编辑、禁用
- 一键切换至指定站点的本地管理系统（SSO）
- 当前 Demo 通过 mock 数据展示站点状态与告警信息；真实环境下需接入监控采集、轮询/WebSocket 和告警通知服务。

**Demo 实现**：
- `/sites` 页面：站点列表 + 详情侧栏
- 站点搜索（名称/编码/机房）
- 状态下拉筛选（全部/在线/降级/离线）
- 新建站点 Dialog
- 同步按钮（更新最后同步时间）
- 禁用/启用切换
- SSO 跳转提示

**未实现**：站点状态实时监控轮询、告警自动触发

#### 4.2.2 统一检索（4.1）

**功能**：
- 跨站点、跨光盘库节点、跨数据中心统一检索
- 支持按文件名称、光盘编号、存储卷、文件类型等维度筛选
- 从检索结果直接发起回迁任务
- 检索结果导出

**Demo 实现**：
- `/search` 页面：高级筛选 + 结果列表 + 分页
- 站点/部门/文件类型/光盘编号/存储卷联动过滤
- 检索按钮 + 重置按钮
- 导出按钮 Toast 提示
- 发起回迁按钮 Toast 提示

**未实现**：千万级文件检索性能、真实文件导出、权限过滤逻辑

#### 4.2.3 统一任务管理（4.2）

**功能**：
- 新建备份/恢复任务
- 任务暂停/恢复/重置
- 数据巡检任务（抽样校验光盘可读性）
- 任务进度监控与提醒
- 任务日志导出

**Demo 实现**：
- `/tasks` 页面：任务列表 + 日志面板
- 任务类型 Tab 切换（全部/备份/恢复/巡检/刻录）

**未实现**：备份中新建恢复的任务依赖控制、真实邮件提醒、自动告警触发

#### 4.2.4 日志管理（5.1）

**功能**：

**Demo 实现**：

### 5.2 未实现（生产级能力缺失）

| 类别 | 缺失内容 | 说明 |
|------|----------|------|
| 登录认证 | 无登录页，无 ADFS/LDAP 集成 | 所有页面直接访问，无权限隔离 |
| 数据持久化 | 无数据库，mock 数据在内存 | 页面刷新后数据恢复初始状态 |
| 后端 API | 无真实 API 接口 | mock 数据无法与真实后端交互 |
| 实时数据 | 无 WebSocket/SSE 轮询 | 服务监控、任务进度等静态展示 |
| 权限控制 | 权限树 Switch 无实际效果 | UI 操作后无权限校验和同步 |
| 邮件通知 | 测试邮件仅 Toast | 无真实邮件发送机制 |
| 文件导出 | 无真实文件生成 | 仅 Toast 提示"导出成功" |
| 数据同步 | 同步按钮仅更新时间戳 | 无与站点系统的真实数据交互 |
| 盘笼移位 | 移位按钮无实际行为 | 无移位登记表单和历史记录 |
| 索引导出 | 无按盘笼导出功能 | 仅有文件列表展示 |

### 5.3 技术边界说明

**当前 Demo 的定位是"可演示的前端原型"，而非生产系统**：

1. **状态管理**：使用 `useState` 管理页面级状态，`store/notification.ts` 使用模块级单例。页面刷新或切换页面后状态重置，不适合需要跨页面保持状态的业务场景。

2. **Mock 数据**：所有数据为 `lib/mock/` 目录下的静态变量，数据量和数据多样性有限（如检索结果仅 8 条）。无法展示大数据量、高并发、复杂查询场景。

3. **类型安全**：TypeScript 类型定义覆盖了 mock 数据，但不覆盖真实 API 响应。后续接入真实 API 时需要重新定义类型。

4. **SSR/SSG**：当前所有页面为静态预渲染（Static），不适合需要实时数据或个性化数据的场景。

## 6. 后续接入真实后端的思路

> ⚠️ 以下为技术预留说明，不代表当前已实现。

### 6.1 API 层接入思路

在 `lib/api/` 目录下创建 API 调用模块：

```typescript
// lib/api/sites.ts
export async function getSites(): Promise<Site[]> {
  const res = await fetch('/api/sites')
  return res.json()
}

export async function createSite(data: CreateSiteInput): Promise<Site> {
  const res = await fetch('/api/sites', {
    method: 'POST',
    body: JSON.stringify(data),
  })
  return res.json()
}
```

### 6.2 状态管理建议

- **页面级状态**：`useState` + `useReducer`（如已足够）
- **跨页面共享**：可引入 Zustand / Jotai / Redux Toolkit
- **服务端数据**：React Query / SWR（推荐，支持缓存、轮询、乐观更新）

### 6.3 数据同步思路

- **站点数据**：轮询 `/api/sites` + WebSocket 推送站点状态变更
- **任务数据**：轮询 `/api/tasks` + WebSocket 推送任务状态变更
- **日志数据**：WebSocket 实时推送，或按需加载（分页）
