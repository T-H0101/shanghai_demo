## 一、文档目标

本文档记录当前 demo 的功能完成状态，以及后续接入真实数据库前后的开发规划。

**核心原则**：
- P0 标记为已完成
- P1 / P2 / P3 作为 backlog，暂不实现
- 每个功能说明业务价值、当前状态、依赖数据、未来实现方式
- 明确哪些功能适合接数据库前做，哪些必须等数据库/后端接口确认后再做

## 三、P1 Backlog

> 优先级高，建议下一迭代实现

### 3.1 介质检测

**业务价值**：通过对光盘和硬盘进行物理层健康检测，在数据损坏前预判介质老化，及时迁移数据，是长期归档的核心保障。

**当前 demo 基础**：无，介质检测功能尚未实现。

**需要哪些数据库表**：
- `media_tests`：检测记录表（test_id, media_id, test_type, test_time, test_result, RSER, BE_Sum, UE, status）
- `media_items`：介质信息表（media_id, type, serial_no, capacity, smart_status, health_score）
- `test_thresholds`：检测阈值配置表

**需要哪些 API**：
- `GET /api/media-tests` - 查询检测记录
- `POST /api/media-tests` - 创建检测任务
- `GET /api/media-tests/:id` - 获取检测详情（含指标曲线数据）
- `GET /api/media-items/:id/health` - 获取介质健康状态

**是否建议接数据库前 mock**：是，可先实现 UI + Mock 数据

**推荐实现时机**：接数据库前可做 UI 框架，接数据库后接入真实 API
