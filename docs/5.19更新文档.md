# 5.19 Demo 优化方向文档

> 本文档基于 requirements.md 审计结果，明确 Demo 优化方向与实施策略。
>
> **核心原则**：不接入真实后端，使用 Mock 虚拟数据，但让数据足够真实、可演示、有说服力。

---

## 一、现状定位

### 1.1 当前 Demo 完成度

| 维度 | 完成度 | 说明 |
|------|--------|------|
| 页面路由 | 100% | 8 个页面 + 登录页，全部可访问 |
| UI 组件 | 100% | 57 个 Radix UI 组件 |
| Mock Authentication | 100% | Site-based 站点权限隔离 |
| 页面交互 | ~70% | Toast 反馈、部分 Dialog、有展示无操作 |
| 数据真实性 | ~40% | 数据量少、场景简单、缺乏业务细节 |
| **综合** | **~75%** | 适合 Demo 演示，但需强化数据真实性 |

### 1.2 Requirements vs Demo 差距

当前 Demo 与 Requirements 的主要差距**不在于功能缺失**（页面都有），而在于：

1. **数据量不足**：仅 6-8 条 Mock 数据，Requirements 要求千万级检索
2. **数据场景简单**：缺乏真实的业务场景（临床试验、影像数据等）
3. **交互深度不够**：只有"展示"，缺少"操作/控制/状态流转"
4. **缺乏时间维度**：无历史记录、无状态变化、无同步时间线

---

## 二、优化策略：让 Mock 数据更真实

### 2.1 数据真实不等于数据量大

真实感的核心是**业务场景的完整性**，而非数据量：

| 要素 | 不真实（当前） | 真实（目标） |
|------|---------------|---------------|
| 时间维度 | 固定时间戳 | 时间线：创建时间、同步时间、操作时间交错 |
| 状态流转 | 静止状态 | 动态状态：运行中→暂停→完成→失败 等多种状态 |
| 数据关联 | 独立数据 | 关联数据：任务关联站点、站点关联设备、设备关联盘笼 |
| 业务场景 | 通用数据 | 行业数据：临床试验编码、影像检查号、存储卷序列号 |
| 操作记录 | 无痕迹 | 有痕迹：操作人、操作时间、操作结果 |

### 2.2 数据关联设计（关键）

让各模块数据形成关联，形成"数据网络"：

```
站点 (6个)
  ↓ 关联
设备 (每站点 2-4 个光盘库)
  ↓ 关联
盘笼 (每个设备 10-20 个槽位)
  ↓ 关联
光盘 (每盘笼含多张光盘)
  ↓ 存储
文件索引 (每光盘含多个文件)
  ↓ 归属
部门 (多个部门)
  ↓ 关联
用户 (部门下有多个用户)
  ↓ 操作
任务 (用户创建/执行任务)
```

**优化后 Mock 数据应体现这种关联性。**

## 三、页面级优化方案

### 3.1 首页仪表盘（高优先级）

**当前问题**：
- 统计数据固定，无业务含义
- 图表数据与业务无关
- 告警中心与实际数据脱节

**优化方向**：
```typescript
// 统计卡片：使用真实业务数据
StatsCards:
  - 全局站点：24个 (在线22 / 负载高1 / 离线1)
  - 今日任务：新增12 / 运行中8 / 完成45 / 失败2
  - 同步状态：最近同步 14:32:05 / 成功 / 同步耗时 8s
  - 异常告警：3条 (严重1 / 警告2)

// 图表：使用真实业务趋势
SyncTrendChart:
  - 7日任务趋势（备份/恢复/巡检）
  - X轴：日期；Y轴：任务数
  - 数据点：每天的任务统计

// 告警中心：与站点数据联动
AlertCenter:
  - 告警等级：严重/警告/提示
  - 告警来源：站点名 + 设备名
  - 告警时间：真实时间（相对当前时间）
```

### 3.2 站点管理页面（中优先级）

**当前问题**：
- 站点信息简单（仅名称/编码/状态）
- 无硬件设备信息
- 无同步历史

**优化方向**：
```typescript
Site 详情面板增加：
  - 硬件信息：光盘库数量、盘笼数量、槽位数量
  - 连接信息：IP、端口、最后连通时间
  - 同步信息：最后同步时间、同步状态、同步耗时
  - 告警信息：当前告警数、严重告警数

站点列表增加：
  - 每站点显示"设备数"和"在线盘笼数"
  - 站点状态带时间：在线(稳定)、负载高(5分钟前)、离线(14:22:05)
```

### 3.3 任务管理页面（高优先级）

**当前问题**：
- 任务类型仅 4 种，但数据简单
- 无任务队列概念
- 无实时日志
- 任务与站点/设备无关联

**优化方向**：
```typescript
任务列表增加：
  - 任务类型：备份/恢复/巡检/刻录，区分颜色和图标
  - 优先级：高/普通，优先任务置顶或特殊标记
  - 进度条：实时进度（模拟 10 秒刷新）
  - 关联信息：站点名、设备名、存储卷

新建任务 Dialog 增加：
  - 任务类型选择（备份/恢复/巡检/刻录）
  - 目标站点选择
  - 目标设备/存储卷选择
  - 优先级选择

任务详情面板增加：
  - 实时日志区域（模拟滚动日志）
  - 操作记录（暂停/恢复/重试的时间和操作人）
```

### 3.4 审计日志页面（中优先级）

**当前问题**：
- 日志类型区分不够
- 无任务类型字段
- 无数字签名标识

**优化方向**：
```typescript
日志列表增加：
  - 任务类型标签：刻录任务/回迁任务/系统任务
  - 操作结果图标：成功(✓) / 失败(✗) / 警告(!)
  - 数字签名状态：有签名验证 / 无签名

日志 Tab 区分：
  - 操作流水：用户操作（登录/登出/新建/修改）
  - 安全日志：认证/授权/权限变更
  - 任务日志：备份/恢复/巡检/刻录任务执行记录
  - 合规报表：定期生成的合规检查报告
  - 告警策略：告警规则变更记录

日志详情面板增加：
  - JSON 格式详情（包含任务类型、设备ID、光盘编号等）
  - 数字签名状态标识
  - 原始数据展开/收起
```

### 3.5 统一检索页面（中优先级）

**当前问题**：
- 仅 8 条 Mock 数据
- 无搜索结果统计

**优化方向**：
```typescript
搜索表单保持：
  - 文件名、站点、部门、文件类型、光盘编号、存储卷

搜索结果增加：
  - 结果统计："共找到 128 条记录，展示了前 20 条"
  - 站点归属标签：每个结果显示所属站点
  - 数据类型标识：原始文件/备份文件/归档文件

结果列表数据更真实：
  - 文件名：临床试验编码_检查日期_序列号.doc
  - 光盘编号：DISC-2024-XXXXX
  - 存储卷：VOL-CLINIC-001
  - 创建时间：2024-03-15 14:32:05
```

### 3.6 盘架管理页面（中优先级）

**当前问题**：
- 无盘笼移位功能
- 槽位状态简单

**优化方向**：
```typescript
盘架列表增加：
  - 每盘架显示：在线盘笼数/离线盘笼数/总槽位数
  - 盘笼列表：可展开查看每个盘笼的详细信息

盘笼详情面板增加：
  - 盘笼状态：在线/离线/移出中
  - 移位历史：移出时间、移入时间、原站点、目标站点、操作人
  - 光盘列表：光盘编号、状态（正常/损坏/待检测）

槽位可视化优化：
  - 槽位状态颜色：空/已占用/故障/待清洁
  - hover 显示光盘详情
```

### 3.7 系统设置页面（低优先级）

**当前问题**：
- 配置项简单
- 无同步策略配置

**优化方向**：
```typescript
同步配置区域增加：
  - 实时同步：启用/禁用（同现状 Switch）
  - 定时同步：每日 02:00 / 06:00 / 22:00
  - 手动同步：触发按钮（同现状）
  - 同步范围：设备信息 / 文件索引 / 权限信息 / 任务信息

告警阈值区域优化：
  - 站点离线阈值：默认 5 分钟
  - 硬件异常阈值：默认 10 分钟
  - 任务超时时间：默认 30 分钟
  - 容量预警阈值：默认 80%

服务监控区域优化：
  - 服务列表：API网关 / 数据同步 / 任务调度 / 告警引擎
  - 状态：运行中(绿) / 异常(红) / 停止(灰)
  - 指标：CPU / 内存 / 响应时间
  - 刷新按钮（模拟刷新）
```

## 六、注意事项

1. **不接入真实后端**：所有数据仍在 Mock 层，不调用真实 API
2. **保持前端 Mock 逻辑**：页面交互仍使用 useState + setState
3. **数据真实性优先**：宁可 6 条真实关联数据，不要 100 条孤立数据
4. **渐进式优化**：按优先级逐个页面优化，不要一次性重构
5. **保持现有 UI 不变**：仅优化数据结构和展示内容

---

## 七、结论

当前 Demo 的核心问题**不是功能缺失，而是数据不够真实**。

通过：
1. 强化数据关联（站点→设备→盘笼→光盘→文件）
2. 增加时间维度（同步时间、操作时间、状态变更时间）
3. 增加状态流转（任务暂停/恢复/失败/完成）
4. 增加业务场景（临床试验、影像数据、存储卷序列号）

可以让 Demo 在不接入真实后端的情况下，**达到 85-90% 的演示效果**。
