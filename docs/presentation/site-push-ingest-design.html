站点推送式同步方案

Sprint 2B.5 设计文档 | 2026-05-30

背景与决策

原方案规划中心库跨城市直连各站点数据库同步数据。经评估存在以下风险：

  • 跨城市延迟 跨城市直连延迟高，影响同步效率
  • 稳定性风险 网络波动可能导致同步中断
  • 网络策略复杂 需要协调各站点防火墙策略

领导决策：采用站点推送式架构，不做中心库跨城市直连。

总体架构

站点系统
→
数据包推送
默认每小时一次
→
Ingest API
/api/ingest/{table}
→
PostgreSQL
小表（tasks/devices/sites/volumes）进入 PostgreSQL

各站点按统一规范导出数据包，通过 HTTP POST 推送至中心平台。
中心平台负责接收、校验、幂等入库和日志记录。

数据分层策略

小表 / 核心元数据

  • 全量快照进入 PostgreSQL
  • tasks（任务表）
  • devices（设备表）
  • sites（站点表）
  • volumes（存储卷表）

数据量不大，变化不频繁，全量快照最简单可靠

大表 / 文件索引

  • 暂不进入中心库
  • tbl_file（文件表）
  • tbl_folder（文件夹表）

数据量大，后续走 Elasticsearch 增量检索

第一版范围

进入同步

  • tasks（任务表）
  • devices（设备表）

已有 mock 数据验证，同步链路已验证

暂缓

  • sites、volumes（字段待确认）
  • permissions（安全风险高）
  • alerts（依赖规则）

接口方案

接收接口

  • POST /api/ingest/tasks — 任务数据接收
  • POST /api/ingest/devices — 设备数据接收

数据包格式

{
  "siteCode": "SH01",
  "sourceTable": "tbl_task",
  "batchId": "SH01-tbl_task-20260530-1000",
  "snapshotAt": "2026-05-30T10:00:00Z",
  "recordCount": 5,
  "records": [...]
}

batchId 用于幂等，snapshotAt 记录站点生成时间，recordCount 用于校验完整性

安全与可靠性

安全保障

  • API Key 认证（每个站点独立 key）
  • 可选 IP 白名单限制
  • 全站 HTTPS
  • API Key 不入库明文

可靠性保障

  • batchId 幂等（重复推送返回成功）
  • payload_hash 校验内容一致性
  • 整批事务写入（成功或回滚）
  • ingest_batch_log 记录全流程
